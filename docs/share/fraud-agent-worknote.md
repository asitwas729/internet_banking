# Fraud Investigation Agent — 작업공유

> **작성자**: 문수현
> **팀**: AXful Bank · 개인 작업공유
> **부제**: Responsibility-based triage → autonomous fraud investigation loop
> **TECH STACK**: LangGraph · Python · pydantic · Anthropic/OpenAI(벤더 중립) · rich(CLI) — 위에 재사용한 자산: Spring Cloud Gateway · JWT · Redis · PostgreSQL(party · FDS · RBAC)

고객계·인증보안계에서 직접 만든 백엔드(party · 인증 · FDS · RBAC)를 **조사 도구로 재배선**해, FDS 알림 한 건을 스스로 조사하는 에이전트를 설계·구현했다. 정렬에서 끝나는 운영 시스템이 아니라, 사건을 파고들어 **"무슨 공격인가"**를 판단한다.

```
정성 가치 → 정량 지표 → 자동화 → 에이전트
```

## ① 책임 등급표 — 정성 가치를 정량 지표로

기존 FDS는 알림을 이상도 점수로 정렬한다. 그러면 "이상하지 않지만 책임이 무거운" 거래 — 사망계좌 30만 원 인출 — 가 점수가 낮아 큐 맨 아래에 묻힌다.

- **정의** — 규정상 책임을 `L4`~`L0` 등급표로 코드화 — 권리자 적격성(사망·후견)=L4, AML=L3, 소비자보호=L1.
- **기준** — "이상도"가 아니라 **"틀리면 누가 배상하나"**로 우선순위를 정의.
- **연결** — 트리아지 정렬과 에이전트 권고 우선순위의 공통 기준으로 재사용.

**결과**
- ✓ 측정되지 않던 정성 가치를 결정 테이블로 옮김
- ✓ 모든 판정에 규정 출처를 부착해 감사 가능

---

## ② 트리아지 — 우선순위 자동화 (에이전트의 입력 필터)

수백 건의 알림에서 "무엇을 먼저 볼지"를 정하는 단계. 정렬은 결정적이고 재현 가능해야 하므로 — 여기는 의도적으로 룰이다.

- **정렬** — 책임 등급으로 3밴드(즉시 · 가중 · L0) 분류, 새 알림마다 실시간 재배열.
- **게이트** — 사망·후견은 코드로 즉시 차단(fail-closed), 이상도 무시.
- **시연** — 작동하는 인터랙티브 콘솔로 큐 재정렬을 시각화.

**결과**
- ✓ 점수가 아니라 책임으로 줄 세움 — 사망계좌(이상도 15)가 거액 송금(이상도 75) 위로
- ✓ 트리아지 최상단이 곧 조사 에이전트의 입력이 되도록 연결

## ③ Investigation Agent — 경쟁 가설 기반 조사 루프 (핵심)

분석가는 사건을 파고들어 "무슨 공격인지" 조사한다. 같은 알림도 디바이스가 평소 것이냐에 따라 보이스피싱이 되기도, 계정탈취가 되기도 한다 — 다음에 무엇을 볼지가 직전 결과에 달려, 고정 워크플로우로는 짤 수 없다.

- **가설** — 공격 시나리오 5종(보이스피싱·계정탈취·자금세탁·내부자·정상)을 동시에 경합 + 부가 태그 3종(머니뮬·조직·신규계좌).
- **계획** — 분별력 매트릭스를 참고해 다음 도구를 LLM이 선택하고, 선택 이유를 매번 로그(설명가능성).
- **루프** — 증거로 가설을 갱신 → 확정·기각·예산 소진까지 재계획(LangGraph state + 조건부 루프).
- **권고** — 가설 + 근거 사슬 + 책임 등급. 동작은 분석가 승인(HITL) + RBAC.

**결과**
- ✓ 고정 워크플로우가 아니라 경쟁 가설 기반 동적 조사 — LangGraph를 빼면 성립하지 않음
- ✓ device 결과에 따라 같은 알림이 다른 조사 경로로 분기(수취망 조사 ↔ 인증 조사)

## ④ 도구 레이어 & 안전장치 — 백엔드 자산 재사용

에이전트의 조사 도구는 직접 만든 은행 백엔드다. **무엇이 실재고 무엇이 목인지 정직하게 구분**하는 것이 이 설계의 핵심 자산이다.

| 구분 | 도구 | 비고 |
|---|---|---|
| **실연결** | `get_auth_events`(CERT_FAIL_BLOCK) · `get_customer` · `get_party`(사망, END_REASON) | 실재 스키마, 읽기 연동 |
| **목** | 후견 · `fds_history` · `device` · `str` · `aml` · `related_accounts` | 별도 인프라 필요, 인터페이스는 실 스키마 기반 |
| **안전** | 결정적 사실=코드(fail-closed) · 동작=HITL+RBAC · 조사 예산 상한 · 타임아웃=부분결과 인계(fail-soft) | |

> 점선 자산(Spring Cloud Gateway · JWT · Redis · PostgreSQL · party · FDS · RBAC) = 고객계·인증보안계에서 만들어 조사 도구로 재사용한 자산.

**결과**
- ✓ 맨땅 토이가 아니라 실재 스키마 기반, 인증 이벤트는 실제로 연결
- ✓ 무엇이 실재고 무엇이 목인지 정확히 구분해 — 과장 없이 설계 경계를 문서화

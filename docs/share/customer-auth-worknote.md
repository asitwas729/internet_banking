# 고객계 · 인증보안계 — 작업공유

> **작성자**: 문수현
> **팀**: AXful Bank
> **분류**: 작업공유 (고객·인증 도메인)
> **TECH STACK**: Spring Cloud Gateway · Spring Security · JWT · Redis · JPA · PostgreSQL · Next.js

**신원 → 정체성 → 권한 → 이상거래 탐지**로 이어지는 흐름을 백엔드에서 설계하고 프런트엔드까지 직접 구현했다.

## ① 인증 아키텍처 — JWT 기반 인증 체계

MSA 환경에서 인증을 각 서비스에 맡기면 검증 로직과 키 관리가 분산된다. 이를 방지하기 위해 **JWT 검증을 게이트웨이로 집중**시키고, 그 위에 권한·세션·인증수단 체계를 구성했다.

- **검증** — JWT를 게이트웨이에서 1회 검증한 뒤 `X-User-*` 헤더로 신원을 전파. 위조 가능성이 있는 헤더는 제거 후 재주입하고, 내부 서비스는 Stateless하게 유지.
- **권한** — 직원의 역할·지점·등급 정보를 토큰에 포함해 내부 서비스가 DB 조회 없이 `hasAnyRole` 기반 인가를 수행.
- **수명** — Refresh Token을 Redis Hash로 관리해 1회용 Rotation과 재사용 탐지를 구현. 세션에는 10분 카운트다운을 두고, 연장 시 토큰 재발급과 타이머·폴링 상태를 동기화.
- **수단** — 인증서 로그인(일련번호 + PIN)과 QR 로그인(PC 폴링 ↔ 모바일 승인)을 구현.

**결과**
- ✓ JWT 검증 로직을 게이트웨이로 일원화
- ✓ Refresh Token Rotation과 재사용 탐지 구현
- ✓ 프런트엔드에서도 401 인터셉터, QR 폴링 상태머신, 세션 카운트다운을 구현해 인증 흐름을 일관 처리

## ② Party 도메인 — 관계자 모델 구현

한 사람이 고객·직원·대리인 등 여러 역할을 동시에 가질 수 있다. 단일 테이블로 관리하면 중복 데이터와 NULL 컬럼이 늘어나므로 **표준 Party 패턴**을 적용했다.

- **저장 구조** — 사람 정보는 `party`에 한 번만 저장하고, 상세 정보는 서브타입 테이블이 `party_id`를 공유 PK로 사용. 역할은 `party_role`로 부여·종료.
- **코드 구조** — JPA 상속·연관관계 대신 `party_id` 식별자 참조 방식으로 연결해 독립 엔티티로 분리.
- **가입 로직** — 본인확인 CI로 기존 Party를 조회, 존재하면 `CUSTOMER` 역할만 추가하고 없으면 신규 Party 생성. 직원 여부는 EXISTS 기반 판별.
- **무결성** — 상태 전이는 엔티티 메서드로 제한, 주민등록번호는 암호화 컬럼으로 저장, Soft Delete로 이력 보존.

**결과**
- ✓ 직원이 고객으로 가입하는 경우에도 기존 Party 재사용 + 역할만 추가
- ✓ 동일 인물의 중복 생성 방지
- ✓ 프런트엔드에서 코드값을 사용자 친화 명칭으로 변환하고 역할 변경 내역을 이력으로 시각화

## ③ RBAC 연동 — 역할 권한을 프런트엔드부터 백엔드까지 일관되게

서비스가 늘어날수록 역할 정의가 분산되면 동일 권한이 서로 다른 이름으로 관리될 수 있다. 이를 막기 위해 **역할 어휘를 하나로 통합**했다.

```
JWT Roles → Gateway → X-User-* Header → hasAnyRole → Frontend AuthGuard
```

- **정의** — 분산된 역할 상수를 `common` 모듈의 `BankRole` Enum으로 통합.
- **표현 정렬** — JWT Claim과 Spring Security의 `ROLE_` Prefix 차이를 Enum 내부 메서드로 흡수.
- **백엔드** — 직원 전용 API(`/api/v1/internal/**`)를 `hasAnyRole` 기반으로 보호.
- **프런트엔드** — `AuthGuard`와 메뉴 권한 제어에 동일한 `BankRole` 사용. 단, 프런트 제어는 UX 목적이며 실제 접근 제어는 백엔드에서 수행.

**결과**
- ✓ JWT 역할 정보가 게이트웨이·백엔드 인가·프런트 권한 제어까지 동일한 `BankRole` 어휘로 연결
- ✓ 하나의 역할 정의를 프런트·백엔드가 공유해 권한 정책 변경 시 수정 지점을 축소

## ④ FDS — 이상거래 탐지

①~③이 정상 사용자의 신원·정체성·권한을 처리하는 순방향 흐름이라면, FDS는 그 위에서 **비정상 패턴을 탐지하는 방어 계층**이다. 인증은 자격 증명 자체뿐 아니라 무차별 대입·반복 실패 같은 비정상 패턴 대응도 필요하므로, 로그인·인증서 사용 이벤트를 룰 기반으로 평가하도록 구성했다.

- **룰 구조** — `fds_rule`에 탐지 조건(JSON)·위험 가중치·동작(`BLOCK`/`CHALLENGE`/`MONITOR`)을 정의. 현재 3종 룰을 운영(V8 시드):
  1. `LOGIN_FAIL_BLOCK_10` — 30분 내 로그인 실패 10회 차단
  2. `CERT_FAIL_BLOCK_5` — 10분 내 인증서 실패 5회 차단
  3. `PWD_CHANGE_MONITOR_3` — 1일 내 비밀번호 3회 변경 모니터링

  탐지 결과는 `fds_detection`, 후속 사고 처리는 `fds_incident`로 분리.
- **평가 연결** — 로그인·인증서 사용 시 `FdsService.evaluate()` 호출. 룰이 정의돼 있어도 호출부가 없으면 동작하지 않는다는 점을 발견해 `CERT_FAIL_BLOCK` 룰을 실제 호출 경로에 연결하고 테스트로 검증.
- **정책 분리** — 실패 경로는 BLOCK 발동 시 즉시 차단(`CUST_060`), 성공 경로는 Silent 평가로 로그만 남김. 정상 사용자를 오탐 차단하지 않기 위한 설계.
- **일원화** — 비밀번호·PIN·인증서·QR 등 로그인 이벤트의 FDS 평가를 공통 후처리(`AuthEventService`)로 모아 경로별로 다른 검증 로직이 생기지 않게 함.

**결과**
- ✓ 인증서 반복 실패 누적 시 자동 차단(`CERT_FAIL_BLOCK`)이 실제 동작하도록 연결 + 회귀 방지 테스트 추가
- ✓ 탐지 정책을 DB 기반 룰로 분리해 코드 수정 없이 룰 추가·변경만으로 대응
- ✓ 실패는 차단, 성공은 모니터링으로 처리해 보안성과 사용자 경험의 균형 고려

> Last updated: 2026-05-20

## 1. 커밋 메시지 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 준수.

### 형식
```
<type>(<scope>): <subject>

<body>           # 선택, "왜"를 설명
<footer>         # 선택, BREAKING CHANGE / 이슈 참조
```

### type
| type | 용도 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변화 없는 구조 변경 |
| `perf` | 성능 개선 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서만 변경 |
| `chore` | 빌드/설정/의존성 |
| `ci` | CI 파이프라인 |

### scope
서비스/모듈명 사용. `customer` / `deposit` / `loan` / `payment` / `common` / `infra` / `buildSrc` / `docs`.

### subject
- **50자 이내**
- 명령형 ("add", "fix", "remove" — 한국어면 "추가", "수정", "삭제")
- 마침표 없음
- 한글/영문 중 하나로 일관성 유지 (이 레포: **한국어 권장**)

### body
- 72자 줄바꿈
- **WHAT이 아니라 WHY**. 코드 diff가 WHAT을 이미 보여준다.

### 예시

✅ Good
```
feat(loan): 대출 신청 한도 검증 로직 추가

기존엔 클라이언트 입력을 그대로 신뢰했으나, 금감원 가이드라인에
따라 서버에서도 DSR 40% 한도를 강제할 필요가 있어 추가.
```

```
fix(payment): 동시 송금 시 잔액 음수 가능한 경쟁 조건 수정
```

```
chore(infra): postgres 컨테이너 4개로 분리 (서비스별 격리)
```

❌ Bad
```
update code              # type/scope/내용 모두 없음
feat: 수정함.            # 무엇을? 마침표 금지
feat(loan): added a brand new feature for loan limit  # 영어/한글 혼용, 길이 초과
```

## 2. 브랜치 네이밍

```
<type>/<scope>-<short-desc>
```

- `feature/loan-dsr-validation`
- `fix/payment-race-condition`
- `chore/infra-prometheus-target`

AI 도구가 작업할 때는 사용자가 지정한 현재 브랜치를 그대로 따른다. 별도 지시가 없으면 새 브랜치를 만들지 않는다.

## 3. PR 규칙

- **제목**: 커밋 메시지와 동일 형식 (`<type>(<scope>): <subject>`)
- **본문**: `.github/pull_request_template.md` 자동 적용
- **크기**: 변경 라인 300줄 미만 권장. 그 이상이면 분할.
- **1 PR = 1 의도**. 무관한 변경 섞지 말 것.
- **자기 리뷰**: 머지 요청 전 본인이 diff 한 번 훑을 것.

## 4. 코드 스타일

### Java
- Java 17 toolchain
- 들여쓰기 4 space, LF 줄끝, UTF-8
- 패키지: `com.bank.<service>.<layer>` — layer ∈ {`controller`, `service`, `repository`, `domain`, `dto`, `config`, `exception`}
- Lombok 정책:
  - ✅ `@Getter`, `@Builder`, `@RequiredArgsConstructor`, `@Slf4j`
  - ⚠️ `@Setter` — JPA 엔티티에선 금지, DTO에서만 제한적 사용
  - ❌ `@Data` (equals/hashCode 폭탄), `@AllArgsConstructor`(생성자 의도 흐려짐)
- DTO·엔티티 분리. 컨트롤러는 엔티티를 직접 반환하지 않는다.

### 주석
- **WHY만 쓴다**. WHAT은 코드가 말한다.
- 변경 사유, 이슈 번호, 호출처 같은 휘발성 정보는 코드 주석이 아니라 **커밋/PR 본문**으로.

## 5. 금지 사항

### 보안
- 비밀번호·토큰·API 키 하드코딩 금지 (반드시 `.env` 또는 secret manager)
- `.env` 커밋 금지 (`.env.sample`만 커밋)
- 운영 DB 접속 명령을 자동화에 포함시키지 말 것

### Git
- `--no-verify`로 hook 우회 금지
- `main` 브랜치 직접 푸시 금지
- 무단 `--force` 푸시 금지 (특히 공유 브랜치)
- 이미 push된 커밋을 `--amend`로 수정 후 force push 금지

### AI 흔적
- 커밋 메시지·PR 본문·코드 주석에 다음 **금지**:
  - AI 모델명/버전 ("by claude-opus-4-7", "GPT-4 generated" 등)
  - AI 세션 링크 (claude.ai/code, chat.openai.com 등)
  - `Co-authored-by: Claude <noreply@anthropic.com>` 류의 가짜 공저자
  - "Generated by AI" 류의 자기 홍보 문구
- **이유**: 코드의 책임은 사람에게 있다. AI 도구는 거들 뿐. 추적이 필요하면 PR 라벨(`ai:claude`)로 충분.

## 6. AI 작업 흐름 (공통 행동 규칙)

모든 AI 도구는 다음을 따른다:

1. **큰 변경 전 계획 확인**: 파일 5개 이상 또는 설계 변경 동반 시, 사용자에게 계획을 먼저 보여주고 승인 받는다.
2. **파괴적 작업 전 명시적 승인**: `rm`, `git reset --hard`, `git push --force`, DB 삭제 — 사용자가 그 세션에서 명시적으로 허락한 경우에만.
3. **루트 원인 우선**: 증상만 가리는 패치(`try/catch`로 에러 삼키기, 테스트 비활성화 등) 금지. 원인을 찾고 보고한 뒤 진행.
4. **사용 안 되는 코드 만들지 말 것**: "나중에 쓸지 모르니" 추가 금지. YAGNI.
5. **모르면 묻는다**: 컨텍스트가 부족하면 추측하지 말고 사용자에게 질문.
6. **검증 후 보고**: 빌드/테스트가 가능한 변경이면 실제로 돌려보고 결과를 보고. 못 돌리는 경우 "검증 못 함"을 명시.

## 10. 작업 단위 (Task Granularity)

- **1 세션 = 1 도메인 + 1 의도**
  - **왜**: 여러 도메인을 한 PR에 섞으면 리뷰가 불가능해지고 롤백 단위가 커진다. AI 에이전트는 컨텍스트 윈도우 안에서 한 가지에 집중해야 품질이 유지된다.
  - **위반 시 증상**: PR 라인 폭증 → 리뷰 누락 → 운영 사고.
- **시작 전 영향 파일 목록 출력**
  - **왜**: 사용자가 사전에 범위를 잡고 중단/축소할 수 있어야 한다. 끝나고 보여주면 늦다.
- **종료 조건: 빌드 통과 + 테스트 통과 + 변경 요약, 셋 다**
  - **왜**: "코드는 짰는데 빌드는 못 돌렸어요"가 가장 흔한 사고 패턴. 셋 중 하나라도 빠지면 미완료로 본다.

## 11. 컨텍스트 수집 (Read-Before-Write)

- **수정 전 반드시 현재 파일을 읽는다**
  - **왜**: LLM은 본 적 없는 파일을 "이럴 것이다"로 채워 넣는 경향이 있다. 실제와 다르면 Edit 실패하거나, 더 나쁘게는 의도와 다른 코드가 들어간다.
- **연관 파일 동시 확인** (Mapper Interface ↔ XML ↔ Domain ↔ Test)
  - **왜**: MyBatis는 인터페이스-XML 이름 매핑이 깨지면 런타임에야 터진다. 컴파일러가 못 잡는다.
- **`common/` 수정 전 4개 서비스 grep 필수**
  - **왜**: `common`은 모든 서비스가 의존. 한 줄 바꿔서 4개 서비스가 다 깨지는 일이 실제로 자주 발생한다.

## 12. 모듈 의존 방향

### 이게 무엇인가
"어떤 패키지가 어떤 패키지를 `import`해도 되는가"의 규칙. 화살표는 "이쪽이 저쪽을 안다(의존한다)"는 뜻이다.

```
controller → service → repository
              ↓
            domain   ← 누구나 의존 가능, 자신은 누구도 모름
              ↑
            common   ← 가장 아래, 누구든 의존 가능

config / exception / dto → 보조 계층, 횡단 사용
```

- `controller`는 `service`를 호출한다. **거꾸로 금지**.
- `repository`(Mapper)는 `domain`을 알지만 `service`/`controller`를 모른다.
- `domain`은 **누구도 모른다** — Spring/MyBatis/Web 어노테이션 일체 금지, 순수 POJO.
- **서비스 간** (loan ↔ payment): Java 코드로 직접 import 금지. HTTP/메시지로만 통신.

### 왜 강제하나
1. **테스트 가능성**: `domain`이 Spring을 모르면 단위 테스트가 `new` 한 줄로 끝난다. Spring 컨텍스트 띄우는 데 60초가 걸리기 시작하면 아무도 테스트 안 짠다.
2. **변경 비용 예측**: 의존이 한 방향이면 "이 클래스 바꿀 때 누가 영향받나"가 명확. 양방향이 생기면 사이클이 만들어지고 영향 분석 자체가 불가능해진다 (스파게티).
3. **서비스 분리 가능성**: 지금은 모놀리식이지만 나중에 진짜 마이크로서비스로 쪼개려면 서비스 간 직접 import가 없어야 한다. 처음부터 막지 않으면 1년 뒤엔 못 쪼갠다.
4. **금융 도메인의 안정성**: 비즈니스 규칙(`domain`)이 기술 변경(JPA→MyBatis, Spring 버전업)에 흔들리면 안 된다. 격리되어 있으면 인프라를 바꿔도 도메인은 그대로다.

### 위반 시 증상
- 단위 테스트가 점점 사라지고 통합 테스트만 남음 → CI 시간 폭증
- "이 메서드 시그니처 바꾸려는데 영향 범위를 모르겠다"
- 서비스 A 빌드가 서비스 B 변경으로 깨짐

## 13. SQL / MyBatis

- **`SELECT *` 금지, 컬럼 명시**
  - **왜**: 컬럼이 추가되면 ResultMap 매핑이 silent하게 깨진다. 또 불필요한 컬럼(특히 암호화 컬럼)까지 끌어와 성능/보안 문제가 생긴다.
- **`#{}` 강제, `${}` 금지**
  - **왜**: `${}`는 문자열 치환 → **SQL Injection 직통**. `#{}`는 PreparedStatement 바인딩으로 안전. `${}`가 정말 필요하면 PR 본문에 사유 명시.
- **동적 쿼리는 `<where>` / `<if>`**
  - **왜**: 문자열 concat 방식은 `WHERE AND ...` 같은 문법 깨짐을 런타임에 발견한다. `<where>`는 첫 `AND`를 자동 제거.
- **N+1 금지**
  - **왜**: 컬렉션 100개당 쿼리 100개 → DB 부하 100배. 실 트래픽에서 터진다. `<collection>` + JOIN으로 한 번에.
- **`@Transactional`은 `service` 계층에서만**
  - **왜**: Controller에 붙이면 HTTP 응답 시간 내내 DB 커넥션이 잡혀 풀이 마른다. Mapper에 붙이면 경계가 너무 좁아져 일관성이 깨진다.

## 14. 테스트

- **피라미드 비율: Unit 60 / Slice 30 / Integration 10**
  - **왜**: Integration만 많으면 CI가 30분 넘어가고 아무도 안 기다린다. Unit이 많아야 빠른 피드백.
- **테스트 DB는 H2** (PostgreSQL 호환 모드: `MODE=PostgreSQL;DATABASE_TO_LOWER=TRUE`)
  - **왜**: 인메모리로 빠르고 격리됨. Postgres 전용 문법(JSONB, ON CONFLICT 등)이 필요한 경우만 별도 검토.
- **금지: 테스트 통과시키려고 프로덕션 로직을 수정**
  - **왜**: 테스트는 명세. 명세에 맞춰 코드를 고치는 게 정상, 반대는 명세를 깎는 행위.
- **`@Disabled`에 이슈번호 주석 필수**
  - **왜**: 사유 없는 비활성 테스트는 영원히 안 돌아간다. 6개월 뒤엔 왜 껐는지 아무도 모른다.
- **Fixture Builder 사용** (`test/.../fixture/`)
  - **왜**: 테스트마다 `new Loan(...)` 길게 적으면, 도메인 시그니처 한 번 바뀔 때 50개 테스트가 동시에 깨진다.

## 15. 예외 처리

- **`catch (Exception e)` 금지, 구체 타입만**
  - **왜**: 광범위 catch는 우리가 잡을 의도가 없던 예외(NullPointer, OutOfMemory)까지 삼킨다. 버그가 로그에서 사라진다.
- **예외 삼키기 금지** (`catch{}`, 또는 로깅만)
  - **왜**: 금융 도메인에서 "조용한 실패"는 최악. 송금이 안 됐는데 성공으로 응답하는 사고로 직결.
- **Controller에서 `try/catch` 금지 → `GlobalExceptionHandler` 위임**
  - **왜**: 에러 응답 포맷이 컨트롤러마다 달라지면 클라이언트가 미친다. 한 곳에서만 변환.
- **비즈니스 예외는 `BusinessException` 상속 + `ErrorCode` 보유**
  - **왜**: 에러코드 체계(`CODE_MASTER`)가 표준화되어야 운영/CS/모니터링이 같은 언어로 말한다.

## 16. 로깅

- **레벨 기준**
  - `ERROR`: 사람을 깨우는 것만. **왜**: 모든 게 ERROR면 알림 피로로 진짜 장애를 놓친다. "새벽 3시에 울려도 되는가?"가 기준.
  - `WARN`: 비정상이지만 처리됨
  - `INFO`: 비즈니스 이벤트 (대출 신청, 송금 완료)
  - `DEBUG`: 개발용
- **민감정보 마스킹** (`Masking` 유틸 경유)
  - **왜**: 평문 로그는 개인정보보호법 위반 + 로그 인덱서(Elastic 등)에 누적되면 사실상 영구 노출.
- **MDC `traceId` 필수**
  - **왜**: 분산 환경에서 traceId 없으면 한 요청의 흐름을 못 따라간다. 장애 원인 분석 불가.

## 17. 에이전트 자기 검증 체크리스트

작업 종료 직전 에이전트가 출력:
- [ ] `./gradlew build` 통과
- [ ] 새 메서드/클래스에 테스트 추가
- [ ] 모듈 의존 방향(§12) 위반 없음
- [ ] `SELECT *`, `${}`, `catch(Exception)`, `@Data` 없음
- [ ] 커밋 메시지·코드 주석에 AI 흔적 없음 (§5)
- [ ] `common/` 수정 시 4개 서비스 빌드 확인
- [ ] 새 `ErrorCode`가 `CODE_MASTER`에 반영됨

**왜**: 사람이 매번 같은 항목 검사하면 빠뜨린다. 에이전트가 출력하면 사용자가 한눈에 빠진 곳을 본다.

## 18. 멀티 에이전트 협업

- **작업 브랜치는 사용자가 지정한 것을 따른다**
  - **왜**: 에이전트가 임의로 새 브랜치를 파면 동시 작업자와의 동기화가 깨지고, 사용자가 어디서 작업했는지 추적이 어렵다.
- **공유 파일(`common/`, 본 가이드, `build.gradle`)은 메인 작업자만 수정**
  - **왜**: 가이드 자체가 에이전트마다 다르게 수정되면 진실의 원천이 사라진다.
- **PR에 도구 라벨** (`ai:claude`, `ai:codex`)
  - **왜**: 사후 분석에 필요. 어느 도구가 어느 영역에서 일을 잘하는지 데이터로 봐야 한다.

## 19. 금융 도메인 특수 규칙

- **금액은 `BigDecimal`, `double`/`float` 금지**
  - **왜**: `0.1 + 0.2 != 0.3`. 부동소수점은 금융에서 즉시 회계 불일치를 만든다.
- **금액 비교는 `compareTo()`, `equals()` 금지**
  - **왜**: `new BigDecimal("1.0").equals(new BigDecimal("1.00"))`는 `false` (scale 다름). 버그 단골.
- **소프트 삭제만, 하드 삭제 금지**
  - **왜**: 금융 데이터는 법정 보관 기간(보통 5~10년)이 있다. `DELETE`는 법 위반.
- **상태 변경은 자동 이력(`STATUS_HISTORY`) 우회 금지**
  - **왜**: 감사 추적성. 누가 언제 어떻게 바꿨는지가 안 남으면 금감원 감사에서 사고.
- **민감 컬럼 암호화** (`CryptoService` 경유)
  - **왜**: 주민번호/계좌번호 평문 저장은 개인정보보호법 위반 + 유출 시 회사가 끝난다.
- **시간은 UTC 저장, KST 표시**
  - **왜**: 서버 이전/DST/멀티 리전 대응. KST로 저장하면 나중에 다 깨진다.
- **날짜 타입은 `LocalDate`/`LocalDateTime`, `Date`/`Calendar` 금지**
  - **왜**: 가변/스레드 안전성 문제. 표준 시간대 처리도 후진다.

## 20. 의존성 추가

- **PR 본문에 사유 명시**
  - **왜**: 라이브러리 하나가 보안 취약점 / 라이선스 / 트랜지티브 의존 폭발의 원인이 된다. 의도적으로 마찰을 준다.
- **GPL 계열 금지**
  - **왜**: 상용 서비스에 GPL을 링크하면 소스 공개 의무가 생긴다. 법무 사고.
- **같은 기능 라이브러리가 이미 있으면 그것 사용** (예: HTTP 클라이언트 통일)
  - **왜**: 중복은 학습 비용·유지보수 비용을 두 배로 만든다.

## 21. 성능 / 리소스

- **트랜잭션 내 외부 API 호출 금지**
  - **왜**: 외부 API가 느리면 DB 커넥션이 그동안 잡혀 풀이 마른다. 한 서비스 장애가 전체로 번진다.
- **TTL 없는 캐시 금지**
  - **왜**: 무한 캐시는 메모리 누수 + stale 데이터 영구화. 둘 다 운영 사고.
- **페이징 없는 `findAll` 금지**
  - **왜**: 개발 단계엔 100건이라 괜찮아 보임. 운영 1년 뒤 100만 건이면 OOM.
- **Redis 키 네이밍**: `{도메인}:{유형}:{id}` (예: `loan:application:12345`)
  - **왜**: 키 네임스페이스가 섞이면 invalidation/조회/디버깅이 다 어려워진다.

# AI 협업 공통 가이드

> Last updated: 2026-05-17
> Scope: 이 레포에서 작업하는 **모든 AI 도구**(Claude Code, Codex, Cursor, Copilot, 기타) 및 사람 기여자.
> 각 도구별 진입 파일(`CLAUDE.md`, `AGENTS.md` 등)은 이 문서를 가리키는 포인터일 뿐, **진실의 원천은 여기**다.

## 1. 커밋 메시지 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 준수.

### 형식
```
<type>(<scope>): <subject>

<body>           # 선택, "왜"를 설명
<footer>         # 선택, BREAKING CHANGE / 이슈 참조
```

### type
| type | 용도 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변화 없는 구조 변경 |
| `perf` | 성능 개선 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서만 변경 |
| `chore` | 빌드/설정/의존성 |
| `ci` | CI 파이프라인 |

### scope
서비스/모듈명 사용. `customer` / `deposit` / `loan` / `payment` / `common` / `infra` / `buildSrc` / `docs`.

### subject
- **50자 이내**
- 명령형 ("add", "fix", "remove" — 한국어면 "추가", "수정", "삭제")
- 마침표 없음
- 한글/영문 중 하나로 일관성 유지 (이 레포: **한국어 권장**)

### body
- 72자 줄바꿈
- **WHAT이 아니라 WHY**. 코드 diff가 WHAT을 이미 보여준다.

### 예시

✅ Good
```
feat(loan): 대출 신청 한도 검증 로직 추가

기존엔 클라이언트 입력을 그대로 신뢰했으나, 금감원 가이드라인에
따라 서버에서도 DSR 40% 한도를 강제할 필요가 있어 추가.
```

```
fix(payment): 동시 송금 시 잔액 음수 가능한 경쟁 조건 수정
```

```
chore(infra): postgres 컨테이너 4개로 분리 (서비스별 격리)
```

❌ Bad
```
update code              # type/scope/내용 모두 없음
feat: 수정함.            # 무엇을? 마침표 금지
feat(loan): added a brand new feature for loan limit  # 영어/한글 혼용, 길이 초과
```

## 2. 브랜치 네이밍

```
<type>/<scope>-<short-desc>
```

- `feature/loan-dsr-validation`
- `fix/payment-race-condition`
- `chore/infra-prometheus-target`

AI 자동 생성 브랜치는 도구 prefix:
- `claude/<...>` — Claude Code
- `codex/<...>` — OpenAI Codex
- `cursor/<...>` — Cursor

---

## 3. PR 규칙

- **제목**: 커밋 메시지와 동일 형식 (`<type>(<scope>): <subject>`)
- **본문**: `.github/pull_request_template.md` 자동 적용
- **크기**: 변경 라인 300줄 미만 권장. 그 이상이면 분할.
- **1 PR = 1 의도**. 무관한 변경 섞지 말 것.
- **자기 리뷰**: 머지 요청 전 본인이 diff 한 번 훑을 것.

## 4. 코드 스타일

### Java
- Java 17 toolchain
- 들여쓰기 4 space, LF 줄끝, UTF-8
- 패키지: `com.bank.<service>.<layer>` — layer ∈ {`controller`, `service`, `repository`, `domain`, `dto`, `config`, `exception`}
- Lombok 정책:
  - ✅ `@Getter`, `@Builder`, `@RequiredArgsConstructor`, `@Slf4j`
  - ⚠️ `@Setter` — JPA 엔티티에선 금지, DTO에서만 제한적 사용
  - ❌ `@Data` (equals/hashCode 폭탄), `@AllArgsConstructor`(생성자 의도 흐려짐)
- DTO·엔티티 분리. 컨트롤러는 엔티티를 직접 반환하지 않는다.

### 주석
- **WHY만 쓴다**. WHAT은 코드가 말한다.
- 변경 사유, 이슈 번호, 호출처 같은 휘발성 정보는 코드 주석이 아니라 **커밋/PR 본문**으로.

## 5. 금지 사항

### 보안
- 비밀번호·토큰·API 키 하드코딩 금지 (반드시 `.env` 또는 secret manager)
- `.env` 커밋 금지 (`.env.sample`만 커밋)
- 운영 DB 접속 명령을 자동화에 포함시키지 말 것

### Git
- `--no-verify`로 hook 우회 금지
- `main` 브랜치 직접 푸시 금지
- 무단 `--force` 푸시 금지 (특히 공유 브랜치)
- 이미 push된 커밋을 `--amend`로 수정 후 force push 금지

### AI 흔적
- 커밋 메시지·PR 본문·코드 주석에 다음 **금지**:
  - AI 모델명/버전 ("by claude-opus-4-7", "GPT-4 generated" 등)
  - AI 세션 링크 (claude.ai/code, chat.openai.com 등)
  - `Co-authored-by: Claude <noreply@anthropic.com>` 류의 가짜 공저자
  - "Generated by AI" 류의 자기 홍보 문구
- **이유**: 코드의 책임은 사람에게 있다. AI 도구는 거들 뿐. 추적이 필요하면 PR 라벨(`ai:claude`)로 충분.

## 6. AI 작업 흐름 (공통 행동 규칙)

모든 AI 도구는 다음을 따른다:

1. **큰 변경 전 계획 확인**: 파일 5개 이상 또는 설계 변경 동반 시, 사용자에게 계획을 먼저 보여주고 승인 받는다.
2. **파괴적 작업 전 명시적 승인**: `rm`, `git reset --hard`, `git push --force`, DB 삭제 — 사용자가 그 세션에서 명시적으로 허락한 경우에만.
3. **루트 원인 우선**: 증상만 가리는 패치(`try/catch`로 에러 삼키기, 테스트 비활성화 등) 금지. 원인을 찾고 보고한 뒤 진행.
4. **사용 안 되는 코드 만들지 말 것**: "나중에 쓸지 모르니" 추가 금지. YAGNI.
5. **모르면 묻는다**: 컨텍스트가 부족하면 추측하지 말고 사용자에게 질문.
6. **검증 후 보고**: 빌드/테스트가 가능한 변경이면 실제로 돌려보고 결과를 보고. 못 돌리는 경우 "검증 못 함"을 명시.

## 7. 파일/디렉토리 규칙

- 신규 디렉토리: kebab-case (`loan-service`, 단 자바 패키지는 그대로)
- 신규 파일: 언어 관습 따름 (Java: PascalCase, YAML/MD: kebab-case)
- 한 파일에 무관한 클래스 여러 개 넣지 말 것 (Java)
- 문서는 `docs/` 하위

## 8. 도구별 진입 파일

각 AI 도구는 자기 진입 파일에서 본 가이드를 1순위로 참조한다.

| 도구 | 진입 파일 | 비고 |
|---|---|---|
| Claude Code | `CLAUDE.md` | 자동 로드 |
| OpenAI Codex / Aider 등 | `AGENTS.md` | 사실상 표준화 진행 중 |
| Cursor | `.cursorrules` (있을 때) | 짧은 포인터만 |
| GitHub Copilot | `.github/copilot-instructions.md` (있을 때) | PR/이슈 생성 시 사용 |

## 9. 가이드 갱신

- 본 문서 변경은 **PR로** 진행. 단독 머지 금지.
- 변경 시 상단의 `Last updated` 갱신.
- 도구가 추가되면 §8 표에 한 줄 추가.

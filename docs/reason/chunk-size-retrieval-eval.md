# RAG 청크 크기·overlap 검색 품질 비교 리포트

- **작성일**: 2026-06-18
- **대상**: advisory-service RAG 청킹 설정(`DocumentIngestionService` `CHUNK_SIZE=800`, `CHUNK_OVERLAP=100`)
- **측정 코드**: `services/advisory-service/src/test/java/com/bank/loan/advisory/rag/ChunkSizeRetrievalEvalTest.java`
- **동결 질의셋**: `services/advisory-service/src/test/resources/chunk-eval/queries.json` (50개)

---

## 1. 요약 (TL;DR)

기존 청크 크기 **800자 / overlap 100**은 경험적으로 정한 값으로, 체계적 비교 없이 사용돼 왔다.
실제 사내 문서로 크기 {400·600·800·1000·1200} × overlap {0·50·100·150} 20개 조합을 OpenAI 임베딩으로 검색해
Recall@k·MRR@k·nDCG@k 를 측정한 결과:

> **권고: 청크 크기를 800 → 1000자로 상향, overlap은 100 유지 (1000/100).**

| 지표 | 현재 800/100 | 권고 1000/100 | 변화 |
|---|---:|---:|---|
| MRR@5 | 0.324 | **0.366** | +0.042 (+13%) |
| nDCG@5 | 0.245 | **0.322** | +0.077 (+31%) |
| Recall@5 | 0.460 | **0.540** | +8pp |
| Recall@10 | 0.560 | **0.740** | +18pp |

- **청크 크기가 지배 변수**, overlap은 2차 변수(진폭 ±0.05 vs 크기 ±0.07~0.12).
- 1000/100이 MRR@5·nDCG@5·Recall@10 **세 지표 모두 1위**(Recall@5만 1200/150에 근소 2위).

---

## 2. 배경

advisory-service의 정책문서 인입 파이프라인은 본문을 **문자 기준 800자, overlap 100자 슬라이딩 윈도우**로
청크 분할한다. 이 값은 "토큰 800 ≒ 문자 1000 수준"이라는 보수적 근사로 경험적으로 설정된 것이며,
청크 크기를 바꿔가며 검색 품질을 비교한 근거 데이터는 없었다. 본 리포트는 그 비교를 체계화한다.

## 3. 방법론

| 항목 | 내용 |
|---|---|
| **코퍼스** | `docs/loan-service-api-spec.md` (실제 사내 문서, 95,321자, 헤딩 섹션 213개) |
| **청커** | 운영 코드 `DocumentIngestionService.splitChunks` 그대로 사용(문자 기반 슬라이딩 윈도우) |
| **임베딩** | OpenAI `text-embedding-3-small` (1536차원), L2 정규화 후 코사인 |
| **검색** | in-memory 전수 코사인(인덱스 영향 배제 — 순수 청킹 효과 측정) |
| **질의** | 엔드포인트 섹션 50개를 LLM(gpt-4o-mini)으로 자연어 질문화 → JSON 동결(재현성) |
| **정답 라벨** | 마크다운 헤딩 섹션 span — 검색된 청크의 char 구간이 정답 섹션 구간과 겹치면 정답 |
| **지표** | Recall@5, MRR@5, nDCG@5, Recall@10 |
| **그리드** | 크기 {400,600,800,1000,1200} × overlap {0,50,100,150} = 20셀 |

질의를 헤딩 문자열 그대로 쓰면 lexical 매칭이 너무 쉬워, 섹션 내용을 알아야 답할 수 있는
자연어 질문으로 LLM 합성하고 파일로 동결했다. 평가 단계는 이 고정 질의셋을 읽으므로 LLM 없이 재현 가능하다.

## 5. 해석

1. **청크 크기가 1차 변수.** 400~600 구간은 답이 경계에서 잘려 품질이 낮고, 1000자 부근에서 정점을 찍는다.
   1200은 Recall@5는 가장 높으나 MRR·nDCG가 1000보다 떨어진다(큰 청크일수록 정답 외 내용이 섞여 순위 변별력↓).
2. **overlap은 2차 변수.** overlap 0은 경계 절단으로 대체로 약간 손해, 150은 청크 수만 늘고(임베딩 비용↑) 이득이 없다.
   100 부근이 무난하며, 크기별 최적 overlap은 50~100에서 진동한다.
3. **튜닝 우선순위**: 크기를 먼저 맞추고(800→1000), overlap은 100 유지 후 필요시 미세조정.

## 6. 권고

- **상향: 청크 크기 800 → 1000자, overlap 100 유지.**
- 적용 위치: `DocumentIngestionService.CHUNK_SIZE = 1000` (overlap 그대로 100).
- 기대 효과(본 문서군): MRR@5 +13%, nDCG@5 +31%, Recall@10 +18pp.
- 재인입 필요: 청크 크기 변경은 기존 인덱스를 무효화하므로, 정책문서·사례 임베딩을 **재청킹·재임베딩(백필)** 해야 한다.

## 7. 한계

- **정답 판정이 "섹션 span 겹침"이라 큰 청크에 기계적으로 유리**한 면이 일부 있다. 다만 순위 품질 지표(MRR·nDCG)도
  1000/100을 가리키므로 결론 방향은 견고하다. 더 엄밀히 하려면 정답을 짧은 "답변 앵커"로 좁혀야 한다.
- **코퍼스가 API 스펙 1종**(구조적·고밀도). FAQ·규정 같은 산문체에서는 곡선이 달라질 수 있어, 일반화하려면 코퍼스 다양화가 필요하다.
- **질의 50개 표본** — 예비 14개 큐레이션 질의셋과 순위 경향이 일치(1000자 우위)해 신뢰도를 높였으나,
  통계적 단정에는 표본 확대(100+)가 바람직하다.
- 임베딩은 OpenAI 호출이라 절대 수치에 미세한 run-to-run 노이즈(±0.003)가 있다. 결론은 셀 간 상대 비교로 판단했다.

## 8. 재현 방법

```powershell
# 키는 루트 .env 의 OPENAI_API_KEY 사용 (또는 환경변수 직접 주입)
$env:OPENAI_API_KEY="sk-..."

# (1) 질의셋 재생성 — 필요할 때만. queries.json 을 새로 동결한다.
./gradlew --no-daemon :services:loan-service:test `
  --tests "com.bank.loan.advisory.rag.ChunkSizeRetrievalEvalTest.질의셋_50개_생성_동결"

# (2) 평가 — 동결 질의셋으로 크기×overlap 그리드 측정(LLM 불필요).
./gradlew --no-daemon --rerun-tasks :services:loan-service:test `
  --tests "com.bank.loan.advisory.rag.ChunkSizeRetrievalEvalTest.청크크기별_검색품질_Recall_MRR_nDCG_비교"
```

- 그리드 조정: 테스트의 `CHUNK_SIZES`, `OVERLAPS`, `GEN_COUNT` 상수.
- 키 없이도 `하니스_배선_오프라인_검증` 은 항상 실행돼 문서/섹션/라벨/오프셋 정합을 검증한다(CI 안전).
- 비용: 생성 ~$0.01, 평가 ~$0.02 (text-embedding-3-small 기준).
